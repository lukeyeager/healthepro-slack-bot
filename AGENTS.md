# Agent Guide

## Project overview

A Go bot that polls the Health-e Pro school menu API (`menus.healthepro.com`) once an hour,
persists the current week's menu to SQLite, and serves it as a static HTML page at `/`.
Deployed via docker-compose.

## Key design decisions

- **Fallback logic:** The API exposes `setting` (current) and `setting_original` (original) for
  each day. When `setting.current_display` is empty (a known operator data-entry bug), the bot falls
  back to `setting_original.current_display`. This is the core reliability improvement over the
  "My School Menus" app.
- **No auth:** The Health-e Pro API is fully public — no API keys needed.
- **Hourly fetch:** On each tick the bot fetches all five days of the *display week* and upserts
  them into the store. Failures are logged and counted in a Prometheus metric; the bot stays up
  and keeps serving the last-good cached data.
- **Week rollover:** On Saturday and Sunday the "display week" is the upcoming Mon–Fri block.
  This logic lives in `internal/week` and is shared by both the scheduler and the HTTP handler.
- **No secrets:** All config lives in `config.yaml`. There is no `.env` file.

## Database (SQLite via `modernc.org/sqlite`)

One row per school day, stored in `./data/menus.db` (bind-mounted into the container).

```sql
CREATE TABLE menus (
    date       TEXT PRIMARY KEY,       -- "2026-04-17"
    fetched_at TEXT NOT NULL,          -- RFC3339 timestamp of most recent fetch/overwrite
    source     TEXT NOT NULL,          -- "current" or "original" (which API field was used)
    items      TEXT NOT NULL           -- JSON array of current_display items
);
```

**Write behavior:** `INSERT OR REPLACE` on every successful fetch. Days with no data (school
closed or date absent from API response) are simply not stored; the HTTP handler shows
"No data yet" for those columns.

## Config shape (`config.yaml`)

Generated by `go run ./cmd/setup`. Shape:

```yaml
org_id: 0      # Health-e Pro organization ID
menu_id: 0     # Health-e Pro menu ID
fetch_cron: "0 * * * *"            # hourly by default
db_path: "/data/menus.db"          # bind-mounted from ./data/ on the host
timezone: "America/Chicago"        # week boundary and cron timezone
http_addr: ":8080"                 # menu web page
metrics_addr: ":9090"              # Prometheus /metrics
```

`site_id` is NOT in the config — it's only used during setup to browse menus.

## API shape

All endpoints are unauthenticated GETs on `https://menus.healthepro.com`.

**Discovery endpoints** (used by `cmd/setup`):

```
GET /api/organizations
  → {"data": [{name, organizations: [{id, name, state}]}]}   # grouped by state

GET /api/organizations/{org_id}/sites/list
  → {"data": [{id, name}]}

GET /api/organizations/{org_id}/sites/{site_id}/menus/
  → {"data": [{id, name, meal_type_id, published_months, ...}]}
```

**Daily menu endpoint** (used by the bot):

```
GET /api/organizations/{org_id}/menus/{menu_id}/year/{YYYY}/month/{M}/date_overwrites
```

Each element of `data[]` has:
- `day` — ISO date string (`"2026-04-17"`)
- `setting` — JSON string; check `current_display[]` first
- `setting_original` — JSON string; fall back to `current_display[]` if `setting` is empty
- `setting.days_off` — array; if non-empty the school is closed (returns nil from FetchMenu)

Items in `current_display` have `type` of `"category"` or `"recipe"`.

## Metrics (Prometheus, port 9090)

- `healthepro_requests_total{status_code}` — per-request HTTP status counter
- `menu_fetch_failures_total` — incremented when a day's fetch fails after all retries
